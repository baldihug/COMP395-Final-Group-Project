### Models Implemented
- dataset.py: NSForcingDataset wrapping the .pt files (x→y vorticity pairs)
- models/fno.py: FNO2d — 4 Fourier layers, modes=12, width=64, ~4.7M params

- evaluate.py: reports rel-L2 and inference throughput (samples/sec)

## Next Steps

### Step 1 — Start the MLflow UI (optional but recommended)
  Run this in a separate terminal and leave it open while training:
    ~/.local/bin/uv run mlflow ui
  Then open http://localhost:5000 in your browser.
  This gives you a live dashboard showing loss curves updating each epoch,
  hyperparameters for each run, and side-by-side comparison of FNO vs U-Net.
  MLflow stores everything locally in the mlruns/ folder — no account or API key needed.

### Step 2 — Train FNO
  ~/.local/bin/uv run python train.py --model fno --epochs 100

  What this does:
  - Loads all 10,000 training samples from data/nsforcing_train_128.pt
  - Trains for 100 epochs using Adam optimizer (lr=1e-3) with cosine LR decay
  - Loss function: relative L2 error — measures prediction error as a fraction of
    the target's magnitude, so it's scale-invariant
  - Every 5 epochs: evaluates on the full 2,000-sample test set and prints both
    train and test loss
  - Saves the checkpoint with the best test loss to checkpoints/fno_best.pt
  - Logs all metrics and hyperparameters to MLflow
  Expected time on RTX 2060: roughly 20-40 minutes for 100 epochs

### Step 3 — Train U-Net
  ~/.local/bin/uv run python train.py --model unet --epochs 100

  Identical setup to FNO. Saves best checkpoint to checkpoints/unet_best.pt.
  U-Net has ~31M parameters vs FNO's ~4.7M, so it may train slightly slower
  per epoch despite being a simpler architecture.
  Expected time on RTX 2060: roughly 30-60 minutes for 100 epochs

### Step 4 — Standard evaluation (accuracy + speed)
  ~/.local/bin/uv run python evaluate.py --model fno  --checkpoint checkpoints/fno_best.pt
  ~/.local/bin/uv run python evaluate.py --model unet --checkpoint checkpoints/unet_best.pt

  What this reports for each model:
  - Test rel-L2: prediction accuracy on the held-out 2,000 test samples
    (lower is better; a random model scores ~1.0, a good model scores <0.1)
  - Throughput: how many samples/sec the model can process at inference time
  - Latency: average ms per sample
  These three numbers are the core quantitative comparison for the writeup.

### Step 5 — Zero-shot super-resolution evaluation (required by project spec)
  ~/.local/bin/uv run python evaluate.py --model fno  --checkpoint checkpoints/fno_best.pt  --super_res
  ~/.local/bin/uv run python evaluate.py --model unet --checkpoint checkpoints/unet_best.pt --super_res

  What this does:
  - Both models were trained exclusively on 128x128 data
  - This test resizes the test inputs and targets to 64x64, 128x128, and 256x256
    using bilinear interpolation, then runs each through the trained model
  - Reports rel-L2 at each resolution — shows how accuracy degrades away from
    the training resolution
  Why FNO is expected to win here:
  - FNO operates in Fourier space, which is resolution-agnostic by design.
    The spectral convolution truncates to 12 modes regardless of input size,
    so the model generalises naturally to unseen resolutions.
  - U-Net uses fixed convolutional kernels and pooling tied to spatial structure,
    so it has no principled reason to generalise to new resolutions.
  Note on methodology: we don't have true 256x256 simulation data, so the 256x256
  ground truth is bilinearly upsampled from 128x128. This is a proxy for real
  super-resolution but is the standard approach when only one resolution is available.
  Mention this limitation in the writeup.
